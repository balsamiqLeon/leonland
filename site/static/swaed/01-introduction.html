1 Introduction

The purpose of this style guide is to facilitate consistency in the user interface (UI) design of Sun's browser-based applications. Note that there also exists a set of guidelines, the Sun.com Web Design Standards (Pacifico) guidelines available at http://www.sun.com/webdesign/index.html, designed primarily for web sites. Although it is clear that the Sun.com Web Design Standards (Pacifico) are designed primarily for web sites, and the Web Application Guidelines (SWAED) are designed for web-based applications, it can, at times, be difficult to determine which set of guidelines to use. For example, a web site can provide access to complex application functionality. For more information about when to use each set of guidelines, see Sun Design Guidelines - Web Style Guides at http://xdesign.sfbay.sun.com/DecisionTree/choosing-swaed-pacifico.html.

The guidelines presented here, in conjunction with the components in the Sun Common UI Component Library (in open source, the Project Woodstock components) using the Sun Theme, enable teams to deliver consistent and recognizable Sun branding for applications bearing the Sun logo. The guidelines and components offer visual designers, usability engineers, and software developers the necessary tools for delivering usable and Sun-branded browser-based applications.

If a company will be putting its own logo on a product we provide them, the user interface should be adjusted to reflect that company's brand.

By following the guidelines outlined here, you can obtain a significant level of consistency both within and among your Sun browser-based applications.

This style guide is intended to be a concise and inclusive set of guidelines for designing browser-based Sun applications. These guidelines are intended for external products, but may be used for internal applications as well.

This style guide cannot:

  • Answer or solve all your application design questions and problems
  • Substitute for the involvement of Human Computer Interaction (HCI) and visual design specialists in your application design and development efforts
  • Substitute for having a user interface review, such as one performed by the User Interface Review Board (UIRB), uirb@sun.com. See the UIRB web site, http://uirb.east.sun.com, to determine if your application is required to be reviewed by the UIRB.

To leverage off of existing standards, these guidelines were based on the Java Look and Feel Design Guidelines (JLF style), where appropriate. These guidelines address the gaps that remain whenever JLF style is inappropriate or not applicable to browser-based applications. The Java Look and Feel Design Guidelines (JLF style) were designed specifically for applications providing non-web-based graphical user interfaces. The JLF style guidelines are available at http://java.sun.com/products/jlf/ .

For writing style, follow the guidelines in the Sun Online Help Writer's Guide available at http://docs.sfbay/app/docs/doc/819-3157 and the Sun Editorial Style Guide available at http://docs.sfbay/app/docs/doc/817-3684. In particular, ensure that your use of the common GUI verbs in the table at http://docs.sfbay/app/docs/doc/817-3684/6mj8t1hcm?a=view conforms to the definitions provided in that table. (See exceptions in 11.1.5 Writing Inline Help.) Additionally, see the Guidelines for User Interface Text at http://reviewme.sfbay.sun.com:8080/UUI/minutes/UI_Text.html which provides guidance specifically for text displayed in the user interface.

These guidelines have been defined with the following goals in mind:

  • Aesthetic visual design
  • Ease of learning and use
  • Support for Sun branding
  • A fluid design that scales gracefully to user preferences and varying browser window sizes
  • Support for several browsers and platforms
  • Accessibility for users with disabilities

Note that the designs specified in this style guide target screen resolutions of 1024 x 768 or greater and the web browsers listed here: https://woodstock.dev.java.net/docs/SupportedBrowers.htm".

Although these guidelines work at screen resolutions under 1024 x 768, and might display reasonably when using non-targeted web browsers, the designs are not optimal when used under these conditions.

Multiple usability evaluations done from March 2001 to the present have informed the designs for navigation, wizards, properties, table manipulation and other user interface elements discussed in these guidelines. Usability testing on the designs to be included in future guidelines will continue, as needed.

Implementing web applications using these guidelines

To produce guidelines-compliant web applications, see Appendix A, Component Library Information.

Because the images in the guidelines are for illustration purposes only, do not use them to obtain color values and font specifications.

The correct colors, font sizes, and styles are provided by the Sun Common UI Component Library (in open source, the Project Woodstock components) and are specified by cascading style sheets appropriate for each browser.

For HTML examples and specifications, see Appendix B, Examples and Specifications. However, Sun recommends building web applications using the Sun Common UI Component Library. (see Appendix A, Component Library Information).

Also, note that product names used in the images in these guidelines do not necessarily reflect actual products and should not be interpreted as such.

For a list of authors and contributors to the Web Application Guidelines, past and present, see contributors.html.

Themes and Images

The components in the Sun Common UI Component Library (in open source, the Project Woodstock components) have been architected such that they can reference style information from an entity called a theme. A theme can be designed to specify such information as colors, text attributes, and component layout information. This information can be used to reflect a company's brand. The Sun Common UI Component Library uses the Sun Theme to convey the Sun brand.

The images in these guidelines reflect the Sun Theme.

1.1 Icons Used in These Guidelines

Throughout this style guide, icons call your attention to the look-and-feel design guidelines and to tips for implementing them.

Indicates required guidelines. These guidelines must be followed when you create your application. (Note that these used to be called requirements (as opposed to 'required guidelines') in earlier releases of this guidelines document.)

Indicates advised guidelines. These guidelines should be followed, if possible, in your application. Sometimes, these guidelines explain when following them is most appropriate or what usability trade-offs exist. (Note that these used to be called recommendations in earlier releases of this guidelines document.)

1.2 Application Structure

No single application structure is clearly superior to the others. Each style has advantages and disadvantages which make it suitable for particular situations. The trade-offs are numerous and complex. Due to this, and the need to support a wide variety of project needs, the guidelines strive to be flexible enough to support a variety of application structures.

1.3 Resources for Application Developers

If you are developing an application, you should use the Sun Common UI Component Library (in open source, the Project Woodstock components), which will produce guidelines-compliant HTML. Components have been implemented for most of the elements covered in these guidelines. See Appendix A, Component Library Information.

These guidelines specify the interface design and behavior for these components and describe how the elements should be used together to ensure a consistent and cohesive end product. The appearance of these designs is controlled through cascading style sheets. These style sheets are, in turn, used by the components.

If your application cannot be built using the component library, see Appendix E, Pixel Specifications to get information about the layout of many of the elements in these guidelines.

1.4 Resources for Application Designers

This section includes information and links to internal web sites that can provide useful details that extend beyond the scope of these guidelines.

In addition, for designs that may need to be extended beyond the basic usage of the guidelines, approvals may be needed. This section identifies these situations and provides information about obtaining the necessary approvals.

1.4.1 Approvals


There may be times when the guidelines are extended and approvals are required by the UIRB (uirb@sun.com). See the UIRB web site, http://uirb.east.sun.com, to determine if your application is required to be reviewed by the UIRB.

Certain legal notices within an application must be approved by Sun's Legal Department. See the next section for details.

1.4.2 Legal


For detailed instructions on how to develop a custom legal notice for a Version page (see 18.1.5 Legal Notice), go to http://sunlegal.central.sun.com/Legal/Legal_Notices. Each development team is responsible for composing its own text. When complete, the legal notice comes in varied lengths for different purposes. The short or long form of the legal notice or proprietary rights notice (PRN) may be used on the Version Page. Select the one that is most appropriate for your application. Note: The text for the legal notice must be approved by a Sun legal representative. To obtain contact information for your division's legal department, go to http://sunlegal.central.sun.com/Legal/Legal_Ops/sunlegalweb/Legal_Directory/legal.dir.html.

1.4.3 Branding


General brand guidelines may be obtained through the brand extranet at: https://brand.sun.com. For information about Sun logos, see: http://logos.sun.com/.

1.4.4 Mastheads


There are a number of ways in which the application masthead may be extended. When extended in these ways, approval from the UIRB, uirb@sun.com, is required. The next three guidelines discuss possibilities for extending the masthead.

In the utility bar of the main window masthead (see 4.1.1 Utility Bar), if providing your own custom buttons aside from the required buttons, you must first contact the UIRB, uirb@sun.com, for approval.

In the User Name, Current Role, and Management Server Information section of the main window masthead (see 4.1.2 User Name, Current Role, and Management Server), before making the user information a link, contact the UIRB, uirb@sun.com, to confirm that the planned application-specific behavior is acceptable.

In the Status Area section of the main window masthead (see 4.1.4 Status Area), if you need to use the status area for an application-specific purpose, you must first contact the UIRB, uirb@sun.com, for approval.

The masthead is designed to display a one-line product name. In the rare situation where a product name needs two lines, follow the guideline below.

In the Product Name section of the main window masthead (see 4.1.3 Product Name), if your product name does not fit onto one line, contact the Sun Web Application Experience Design (SWAED) core team at swaed-core@sun.com.

1.4.5 Content Page Titles and Other UI Text

For writing style for content page titles, constructing links, and writing other text for the UI, follow the guidelines in the Sun Editorial Style Guide available at http://docs.sfbay/app/docs/doc/817-3684. In particular, ensure that your use of the common GUI verbs in the table at http://docs.sfbay/app/docs/doc/817-3684/6mj8t1hcm?a=view conforms to the definitions provided in that table. (See exceptions in 11.1.5 Writing Inline Help.) Additionally, see the Guidelines for User Interface Text at http://infobot.sfbay/~pegs/ui_text_style.txt which provides guidance specifically for text displayed in the user interface.

1.4.7 Accessibility


Besides enabling users with disabilities to use your product, section 508 of the Federal Rehabilitation Act mandates that all products used by U.S. government agencies must provide "comparable access" for people with disabilities.

The types of disabilities that should be considered include visual, hearing, physical, and cognitive.

For more information on Section 508 requirements, assistive technologies, and links to technical resources, see this internal Sun site: Sun-wide Accessibility Program Office at http://accessibility.eng.sun.com

To access User Friendlier 508, go to
http://solaris.eng.sun.com/accessibility/508_friendly/index.html
For Labels/Form element information see: http://solaris.eng.sun.com/accessibility/508_friendly/preamble.html#1194.22n.
For Skip Navigation information see: http://solaris.eng.sun.com/accessibility/508_friendly/preamble.html#1194.22o.


The following URL provides a pointer to guidance being provided to Sun technical writers for how to construct a useful description of a nontext element. The general guidance it provides is useful because it provides the application developer with enough guidance to extrapolate on the type of information to include in the descriptions for the application's nontext elements.
More information on writing useful alternate descriptions can be found at http://infobot.eng.sun.com/infodev/Teams/accessibility/sun/doc/merged_508.pdf.

Data tables are a commonly used user interface element for conveying information to users. For more information on how to use table attributes, go to http://accessibility.eng.sun.com/site.topics/writer/tableguidelines.html.

All Sun products must ship with documentation that meets Sun's 508 Documentation policy.  The policy identifies what requirements must be met to ensure the application is providing accessible HTML or accessible PDF documentation. Sun's 508 Documentation policy is at http://accessibility.eng.sun.com/site.topics/section508/sun-policy.html#bigdoc.

For web page accessibility checking, the tool InFocus is very useful. This is a web-based tool for which the APO has purchased a site license. The tool was deployed in early March 2003 and is available for use by any Sun employee. A pointer to InFocus is provided on the APO's web site at http://accessibility.eng.sun.com.

For more information see Appendix D, Accessibility Requirements.

1.4.8 Globalization


Globalization refers to the combined tasks of making an application internationalized (suitable for the global marketplace) and localized (customized for a particular language or region).

The goal of globalizing software is to make it so that code changes are not needed in order to localize it. This reduces development costs when creating software for a global market.

Sun has a Big Rule whose intent is to provide a unifying SMI vision for product internationalization and localization. The ultimate goal is to enable the delivery of fully localized products to our international customers. This rule is considered important enough by the Solaris PAC that exceptions must be approved by the CEO. The full text of the Big Rule (along with the schedule) is available at http://global.eng/Compliance/i18nl10nbigrules.html.

Sun's internal globalization site is: http://global.eng.sun.com, and it includes a compliance matrix http://global.eng.sun.com/compliance.

For more information see Appendix H, Globalization.
