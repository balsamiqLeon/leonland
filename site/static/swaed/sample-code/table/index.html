Overview

This is the sample HTML for the action table component described in Chapter 9, Tables. The guidelines chapter for tables is the definitive source for details on the appearance and behavior of tables and will not be repeated in this document. The release notes at the end of this document contain HTML & css implementation issues and details.

There are two variations on the table's appearance, a standard look and a "lite" look intended for use in e.g., portal pages, where the standard appearance is too visually heavy. This is implemented using contextual selectors, so to view a table in its lightweight mode one only needs to add "TblLt" to the class specification on the table tag, i.e., class="Tbl TblLt". Examples of both variations can be seen in the examples section below.

Release Notes

The fact that tables are specified to take 100% of the width of the window, less 10px margins on either side, forces a platform solution. It turns out that if the 10px margin is specified in a <div class="TblMgn"> IE will actually add the 10px margins to the left and right of the table after it has stretched the table to 100%, which forces horizontal scrolling. To get around this, the style sheet specifies 10px left and right white borders on the table, instead of margins. In the NS6 style sheet, these are margins.

Column Spacing

The fact that the tables need to be 100% of the width of the window means that application developers will need some way of controlling the relative widths of, and be able to specify <nowrap> on, specific columns to ensure that the tables end up with an attractive appearance. In the case of tables with only two data columns, a spacer column is placed to the right of the two data columns in order to take up the extra space, thus allowing the contents of two data columns to be close together. This column uses the styles TblTdSpc and TblTdSpcLst.

Column Alignment and Wrapping of Headers

Columns should be left, center, or right aligned according the the rules specified in the guidelines. There can be no mixed alignment within a column. Testing revealed that the most reliable cross-platform way of getting left or center alignment to work was via the use of the align attribute in the td tag. By default, cell contents for td's are left-aligned, so there is no need to do anything for these cells. th's, on the other hand are center-aligned by default, so it will be necessary to include a align="left" for those cells.

To support attractive wrapping, column headers with sort icons are placed inside of table with the column header text and the sort icon placed in separate cells. In this scheme the text can wrap to two lines and the sort icon will stay bottom aligned and to the right

Dynamic Behavior

The table is a place where the dynamic enabling and disabling of buttons and menus (and their menu items) is likely to be used quite a bit. Typically, this dynamic behavior would be in response to the user checking the checkboxes or radio buttons in the first column of the table, if present. In some cases, certain buttons and menu items should be disabled when the user enters the page (i.e., because nothing has been selected). Because the dynamic enabling and disabling of individual menu items does not work in Internet Explorer, selecting an item that is "disabled" should launch a pop-up window containing a message about why the control has been disabled. There are several default messages described in the Tables chapter of the guidelines, but there may be a need for application developers to specify a custom message.

Accessibility

There are several accessibility features built into the table. The first is that the table title is implemented using the <caption> tag. The caption has a style defined that gives it the appropriate appearance for all browsers.

The table also specifies the use of the "scope" attribute to indicate row and column headers. Tables will always have column headers specified via inclusion of scope="col" in the <th> tags for the table. The exceptions to this are the columns containing checkboxes or radio buttons and the spacer columns used for two column tables. Row headers will exist in most, but not all tables via the inclusion of scope="row" in all of the cells in a column that are identified to contain header information. These row headers also need to be specified with a <th> rather than a <td> tag. For example, if a table has a column containing user names and to the left is a column of checkboxes and to the right are columns containing attributes of those users, then the contents of the cells in the user name column are acting as row headers and would therefore be specified via a th with a scope="row" contained within. On the other hand, if a table contains a system log with time stamp and log entry columns, then neither column should have scope="row" specified in the rows because neither is acting as a row header in this case -- this table would still, however have scope="col" specified in the column headers.

Alarm Cells

When there is an alarm in a table cell, the cell gets a yellow background. Depending on where the cell is in the table (i.e., in a sort column, in the first column, in the last row, etc.), the style for the alarm cell varies somewhat.
