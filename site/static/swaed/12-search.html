12 Search

Search is a tool that can help a user locate a physical or logical object or objects in an application's object hierarchy. Search can be particularly useful when it is the only way to easily create a list of objects that exist across an application's structural or navigational groupings (such as tabs or tree nodes).

The following are different types of search:

  • Quick Search does not require a user to open a secondary-content page to enter a query and may be global or local in scope.
  • Advanced Search requires a user to open a secondary-content page to enter a query and may offer complex scope. Scope is the area or areas of the application searched. Complex scope is described in 12.2.2 Adding Scoping to Advanced Search.

All types of search yield zero or more search results (and zero results can itself be important information). The search results table is never drawn until one or more search results are created (an empty table does not appear ahead).

Search should not be confused with other tools that have related or similar functionality, such as filter, view, and find.

  • Filter is a tool used to draw out a subset from an existing group of objects based on inclusive criteria, and does not substantially change the format of the original object presentation. See 12.1.2 Local Quick Search and 9.5.1 Table Filtering for more details.
  • View is a tool used to specify a substantially different presentation of an object set (such as in a tree, topological view, or list view). For more information, see 6.4 Content Area Navigation.
  • Find is a tool offered by browsers to locate an instance of a word or words only in the contents of a browser window that is currently open and active.

In all search variants, when something is defaulted into the search text field, whether the word Search, an asterix, or the previous search, enable the text field to select the field contents upon taking user focus, and enable typing to replace the existing text.

12.1 Quick Search

There are two types of quick search:

  • Global, a search that can search the entire application.
  • Local, a search that is confined to a specific area of the application, such as a tab.

Provide quick search in all applications that require a user to find a logical or physical object or objects in the application's object hierarchy without opening a secondary-content page.

Provide only one type of quick search in an application.

Two or more quick search tools should not appear in one page view. Such a design would confuse the user because the respective scope and function of these identical-looking search tools would be ambiguous. This ambiguity could also undermine the important relationship that the location the search tool has with its scope. In other words, a product should not offer local quick search in the content area (see 12.1.2 Local Quick Search) if there is a global quick search in the masthead (see 12.1.1 Global Quick Search).

If a product needs to offer quick search with both global and local scopes, provide one search tool in the masthead that accommodates those various scoping needs by using the global quick search variation that includes a drop-down list box for scope (see 12.1.1.3 Global Quick Search: Scoped Text Search).

Show all quick search tools with a Search button, but the button's appearance will vary based on whether it is in the masthead or not, as detailed below.

Make the Search button the right-most item among the quick search elements.

12.1.1 Global Quick Search

Global quick search is a tool whose scope is the application's entire object space. If a user searches using this tool, the entire object space can be searched.

Provide global quick search in the main window masthead of applications that require a user to find something in the application's entire object space.

Place global quick search in a main window masthead directly to the left of the right-aligned global links in the utility bar.

12.1.1.1 Global Quick Search: Predefined Queries

This variation includes just a dropdown list, with no text field. The guidelines in this section only apply to that variation.

Use this variation when you want to restrict the user to obtaining only general, predefined search results.

In the drop-down list box, provide predefined queries or information buckets or object categories that might not be explicitly provided by the application's navigation structure.

In this Predefined Queries option, make the default state (the first item) in the drop-down list box be just the label "- Search -." Provide "Search All" only when there is a small set of possible values, and never default the list to "Search All", as that could lead to accidently doing a very slow query of all possible items, potentially across multiple data sets.

If a user clicks the Search button with the default value "- Search -" in the drop-down list box, show the following error alert message in a JavaScript pop-up window: "Select your search query."

Global Quick Search: Predefined Queries is conceptually and functionally similar to 9.5.1.1 Basic Table Filtering in that it features predefined criteria. However Global Quick Search: Predefined Queries is usable across the application (not limited to a single table on a single page) and presents its results on a new, dedicated results page (instead of just limiting what is shown on an otherwise unchanged page, as happens with Filters).

12.1.1.2 Global Quick Search: Application-Wide Text Search

This variation includes just a text field, with no dropdown list. The guidelines in this section only apply to that variation.

Use this variation if a user might need to look for an object anywhere in the application's object space using a text query.

To save space in the masthead, default the word "Search" in the search text field to provide the labeling instead of having a button with the label "Search", and use the special masthead iconic button as shown in the example above. This loses the ability for the user to see whether wildcards are allowed or not ahead, but is needed to provide some sort of search labelling in this one version which has no label or dropdown to do so otherwise.

If a user clicks the Search button with the text field either empty or containing the default value "Search", show the following error alert message in a JavaScript pop-up window: "Enter your search query."

Provide wildcard functionality in the text field, if needed. Avoid implementing implicit wildcard functionality because this functionality prevents the user from entering exact queries.

12.1.1.3 Global Quick Search: Scoped Text Search

This variation includes both a text field and a dropdown list. The guidelines in this section only apply to that variation.

Use this variation if a user might need to look for an object using a text query, where the scope of objects to be searched can optionally be restricted to a subset of the application's object space, or when both local and global quick-searching is offered.

In the drop-down list, provide predefined information buckets or object categories that might not be explicitly provided by the application's navigation structure.

If this feature is being used to accommodate for local and global quick-searching, provide options to accommodate both local and global scopes.

In a quick search tool that provides both global and local scopes, the drop-down list box menu items should represent existing object groupings offered by the application.

Make the default state (the first item) in the drop-down list box be "Search All", if it is supported. Otherwise, make the default state be just the label "- Search -".

As there is a dropdown next to the text field which already says "Search All" or "– Search –", there is no need to put Search by default into the text field.

Provide wildcard functionality, if needed. Make it explicit by showing an asterisk in the text field by default. Avoid implementing implicit wildcard functionality because this functionality prevents the user from entering exact queries.

If implicit or no wildcard functionality is provided, leave the text field blank by default.

If a user clicks the Search button with the default value "- Search -" in the drop-down list box or with the text field empty, show one of the following error alerts, in a JavaScript pop-up window, depending on what was not specified:

  • "Enter your search query in the text field"
  • "Select an item in the drop-down list box" or
  • "Enter your search query in the text field and select an item in the drop-down list box"

(In this case, * is allowed as the value in the search field, as the dropdown will already constrain the search.)

12.1.2 Local Quick Search

Local quick search limits its scope to a portion of the application's object space corresponding to the place in which it appears. It never appears in a masthead.

For example, in applications with a hierarchical object presentation, when Local Quick Search appears on a page corresponding to a branch of the hierarchy, local searches on those pages are always limited to that branch and possibly below.

Provide local quick search in the content area if a user might want to search only that area and when global quick search is not offered.

When search is provided in the content area (outside the masthead) of a page, label the button with the text "Search" (see 7.1.6 Button Labels ), and make the search button a secondary button (see 7.1.2 Secondary Buttons)

Use local quick search instead of filtering if the collection of objects resulting from the search is so diverse that it does not make sense to display them in an existing table.

Basic filtering should be used when the collection of objects to be filtered can be displayed inside an existing table. This would typically be the case when the table already contained all the objects under consideration but was too large to comfortably view a specific subset. See 9.5.1 Table Filtering for more details.

In applications with a hierarchical object presentation, local quick search should not search upward in the object hierarchy.

Place local quick search directly to the right of left-aligned page actions, if any. If there are no left-aligned page actions, make local quick search the left most element in the local actions bar.

Provide at least 15 pixels of white space to the right of the local quick Search button, to separate it from other content in that same line.

12.2 Advanced Search

Advanced search is a superset of simple search. While the user gives up the convenience of quick search (advanced search requires a user to open a secondary-content page), advanced search offers the user a way to enter a detailed query for more accurate search results. The user may also be offered more guidance on how to use search effectively. For example, help about query syntax.

Provide advanced search in all applications where the user might need to locate objects by entering complex queries that require more than just a drop-down list box and/or a text field.

Here are the three types of advanced search:

  • Global
  • Global with scoping
  • Contextual

Provide only one type of advanced search in an application.

Make advanced search a button in the masthead utility bar, labeled:

  • "Advanced Search..." if the application also offers global quick search or local quick search on any of the application's pages.
  • "Search..." if there is no global quick search, and none of the application's pages offers local quick search.

Place the Advanced Search button in the main window masthead and make its default scope global.

Place the Advanced Search button in the right-hand-side cluster of masthead utility bar buttons, to the left of all other buttons, per these rules:

  • If Global Quick Search is also available in the masthead, render the button as a separate button, separated from the global quick search with a second utility separator:
  • If no Global Quick Search is offered in the application, render the button as the leftmost button within the utility bar button cluster:

When the Advanced Search button is clicked, open an Advanced Search dialog page in either of the following:

  • Replacing the main window (page swap)
  • Opening a pop-up window

Offer one behavior consistently throughout the application when the Advanced Search button is clicked.

For example, advanced search may not open in the main window in one area of the application and then open in a pop-up window in another. One behavior must be used throughout the application.

If an Advanced Search pop-up window is already open, but the user clicks again on the Advanced Search button in the main window, then surface and reuse the Advanced Search window that is already open.

Do not open an additional window or swap the main window. Merely surface the Advanced Search window that is already open, leaving its contents intact. Do not reset any values that remain from previous use.

If advanced search appears by default in the main window, then all object hierarchy context must disappear from the content area (for example, the tabs) to clarify that the user is in a dialog page.

Advanced Searchs must follow the guidelines for secondary-content pages (see 3.8 Secondary-Content Pages).

Make the title of all Advanced Search secondary-content pages clearly show the scope of the search by following the secondary-content page titling convention: "Main Focus of the Page - Object Name."

For example, the content page title of global advanced search in SunPlex Manager(TM) should be "Advanced Search - Business Cluster" where Business Cluster happens to be the highest level of the object hierarchy in SunPlex Manager.

If implicit or no wildcard functionality is provided in any of the text fields, show nothing in those text field by default.

  • "Enter your search query in the text field"
  • "Select an item in the drop-down list box" or
  • "Enter your search query in the text field and select an item in the drop-down list box"

If contextual advanced search is provided, global advanced search cannot be offered with the current design constraints. In most cases, if implementation of contextual advanced search is warranted, the application's object groupings should be heterogeneous enough so that the need to do a global advanced search is minimal.

Contextual advanced search should be used only when the user benefits significantly from having context-specific elements rather than generic elements in advanced search.

Design issues regarding global advanced search and contextual advanced search are still being explored.

12.2.1.1 Global Advanced Search in Main Windows

In applications with Global Advanced Search where window management might be a problem, make Advanced Searchs open in the main window.

There are advantages and disadvantages to making Advanced Searchs open in main windows as opposed to in pop-up windows. The product team must make the appropriate decision regarding which model will best fit user needs. In the design using main windows, the user's current page is replaced with an Advanced Search. The advantages here: that the desktop will not be cluttered with more windows (lessening the likelihood of orphaned windows) and online help on advanced search can be offered conventionally through the Help link in the masthead.

In applications with global advanced search where users are likely to use the search results in tasks that are primary rather than secondary in nature, make Advanced Searchs open in the main window. 

Search is usually considered an ancillary task. However, some applications may use search results in complex ways that relate more to primary tasks. For instance, users usually do a search in a secondary-content page, get results, examine some of those results, and go back to what they were doing before they began the search. However, sometimes users use search results in complex ways that tie closely into what they were doing before initiating the search, such as defining nontransient groups out of objects found in the search results.

When Advanced Search opens in a main window, reiterate the default scope of the advanced search, if any scope limitations exist, in the page inline help.

Provide wildcard functionality in the text field, if needed. Make it explicit by showing an asterisk in the text field by default.

12.2.1.2 Global Advanced Search in Pop-Up Windows

 In applications with global advanced search where modality and context take priority over window management issues, make Advanced Search dialog pages open in a pop-up window.

There are advantages and disadvantages to making Advanced Searchs appear in pop-up windows as opposed to in main windows. The product team must make the appropriate decision regarding which model will best fit user needs. If the dialog page opens in a pop-up window, the advantages are that the user does not lose her primary task's context while doing the search, a left pane dedicated to contextual help can be made available, and the dialog page is modeless.

In applications with Global Advanced Search where users are likely to use search as a secondary task make Advanced Searchs open in a pop-up window.

If Advanced Search opens in a pop-up window, provide left-pane help.

If left-pane help is not provided, consider providing inline help on the page instead, possibly including a link to online help (opening the Help window) where detailed help is provided.

When Advanced Search opens in a pop-up window, reiterate the default scope of the advanced search in the left-pane help, or if left-pane help is not offered, in the page inline help.

Provide wildcard functionality in the text field, if needed. Make it explicit by showing an asterisk in the text field by default.

In the help provided, try to clarify the uses of syntax equivalents in search elements.

12.2.2 Adding Scoping to Advanced Search

Provide scoping in the global Advanced Searchs of all applications where users might need to exclude some area or areas of the application from the search.

To provide simple scoping, use a drop-down list box. Make the first (default) item "All" to represent global scope. Make the rest of the items represent local scopes.

Complex scoping allows users to have more control over excluding areas of the application from the search. For example, users can do the search in three out of six local areas of the application rather than just pick one or all areas.

12.2.3 Contextual Advanced Search

Contextual advanced search is a special type of advanced search in that it is purely local in scope. In addition, contextual advanced search offers very context-specific elements in addition to generic search elements. An example of a context-specific element is a drop-down list box that allows the user to specify a particular alarm severity level.

Provide contextual advanced search in applications where the need for context-specific searching outweighs the need for global advanced search.

Do not use contextual advanced search where the context is limited to a single table on a single page, and where results are best shown in that same table on that same page. For such cases, use 9.5.1.2 Custom Table Filtering instead.

The current design does not allow for the use of contextual advanced search and global advanced search in the same application.

Design issues regarding global advanced search and contextual advanced search are still being explored.

When Advanced Search opens in a main window, reiterate the default scope of the advanced search, if any scope limitations exist, in the page inline help.

When Advanced Search opens in a pop-up window, reiterate the default scope of the advanced search in the left-pane help, or if left-pane help is not offered, in the page inline help.

Here is an example of a local Advanced Search in a pop-up window. Note the use of contextual search elements.

12.3 Search Results

Show the results of a quick search in a main window or in a pop-up window.

Show the results of an advanced search in the same window in which the search query was entered.

For instance, the results of an advanced search in a pop-up window must reload that Advanced Search with search results in the same pop-up window.

When the user clicks on one of the search results, change to that object being displayed in the main browser window and bring that window to the front. However, the search popup remains open in the background for further use until closed.

If search results are displayed in a pop-up window, but the user clicks on a Search button again in a main window, surface and reuse the pop-up window that is already open.

Search results pages must follow the guidelines for secondary-content pages (see 3.8 Secondary-Content Pages).

Here is an example of a global quick Search Results page in a main window:

Make the final scope of the search clear in the content page title of the search results secondary-content page.

Label the full search query "For Search Criteria:"

Display the full query of the search using the syntax of the application's particular search engine. Separate each query item with a comma and a space.

In a Search Query, here are some commonly used syntactic rules, taken from Google:

  • Quotation marks are commonly used to denote exact phrase matches.
  • Plus signs are used to denote required search terms.
  • Vertical bars are used to denote the logical OR.
  • Minus signs are used to denote matches to omit.

Provide search results in a sortable table, and follow the guidelines for tables (see 9.5.2 Table Sorting and 9.1 General Table Guidelines).

If no results are produced from the user's search, provide the following text in the empty table: "The search found no results. Try again by modifying your search." See 9.1.3 Displaying Empty Tables.

Follow table guidelines regarding pagination, titles, etc.

For searches yielding objects in an object-oriented design, label the first column of the results table "Name." In this column, include links to the matching objects' detail pages.

12.3.3 Advanced Search Results

When the user does an advanced search, show the search results as illustrated in this section.

Restore the user's query within the advanced search elements when re-rendering the page.

The search elements do not technically constitute a section in the advanced Search Results secondary-content page. The search elements are offered as a convenience for the user, and do not carry the same weight as the search results table. The search elements may be used if the user would like to do another search or to modify the current search without going to another window.

The search elements are also clearly separated from the search results table with a marked anchor "Back to Top" so that the user can easily jump to the search results should she want to reuse them.

Provide jump-to links to the search components directly under the page inline help, left-align it, and label it "Modify Search."

Provide page inline help if useful for the advanced search elements; use this help to clarify that the user may modify the current search.

Provide "Search", "Reset", and either "Close" or "Back to <Short Page Name>" buttons at the bottom of the page.

Make the Search button a primary button. Make the Reset button and the Close or Back to <Short Page Name> button secondary buttons (see 7.1 Action Buttons). Note that this is different from non-advanced search windows, where the only button at the bottom is Back or Close, and therefore Back or Close is the primary button.
