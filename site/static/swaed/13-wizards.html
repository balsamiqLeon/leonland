13 Wizards

This chapter has separate sections for anatomy (13.1) and behavior (13.2).

A wizard is a series of two or more secondary-content pages, each of which corresponds to a step or portion of a step in a user task. A wizard leads the user through a task one step at a time, requesting a series of responses and finally executing the task based on those responses. Except for user responses, a wizard provides all the information needed to do the task. A wizard's power comes from its dynamic nature: its ability to present different information depending on what is entered by the user in the preceding step. Wizards also expedite complex tasks by grouping steps into a single place.

For sequences of two or more user input steps, use a wizard designed according to the guidelines in this chapter. (The Review Selections and View Results steps do not count as "user input" steps.)

Note: The guidelines in this chapter are for popup wizards only.  Guidelines for inline wizards are currently being developed and will be available in a later release.  If you believe your project requires an inline wizard, it must conform to the following three guidelines which will be included in an upcoming version. 

 Inline wizards may only be used for one or two steps.  Anything requiring more than two steps should use the regular popup wizard. 

 Buttons on the first page must be Next and Cancel, while buttons on the second page must be Previous, Finish and Cancel. 

 Subtitles should read "My Step Title (Step 1 of 2)" or "My Step Title (Step 2 of 2)."

13.1 Anatomy

All wizards have the same basic anatomy. Here is an example picture:

 The left pane contains Help and Steps tabs.  The Steps tab must be the active (front, foreground) tab by default.  

If there is no Help tab then display the Steps in a non-tabbed format.  (Do not display a single Steps tab.)  See 11.3 Left-Pane Help for information on the look and feel of a single pane window. 

 Steps must always be displayed in the Steps/Help Pane, regardless of the number of steps in the wizard.  

The current step's contents appear in the right pane, which is referred to as the content pane.  The topmost element in the content pane is a subtitle uniquely identifying the step and its purpose.  The contents of the current wizard step appears below the wizard subtitle.

Format the content pane's subtitle as follows: "Step <step number>: <current step's text as shown in the steps tab>". For example, if the step is shown as "1.1 Specify QFS Type", the content pane's subtitle would be: "Step 1.1: Specify QFS Type". Although the format changes slightly, the step number and step text remain exactly the same between the steps tab and the content pane. Provide step instructions under the wizard subtitle to give a simple direction for completing the current step. Note that detailed help information is to be placed on the help tab in the left pane.

In the example picture above, the Previous button is disabled because the current step is the first step in the wizard.  Likewise, the last step has a Finish button instead of a Next button.  Middle steps of any wizard display Previous, Next, and Cancel buttons.  Follow the guidelines for those buttons, in 7.1 Action Buttons.

 The content area within the wizard should be scrollable, if necessary. The height and width of both the left and right panes are set to a percentage of the entire window's height and width. If the content of either pane does not fit in the allotted space, horizontal and/or vertical scrollbars should appear. The scrollbars should appear in the left and/or right panes as needed.  Scroll bars should not appear if the content displays within the default size of the pane.

 The following list describes the sizing of the wizard window:

  • The default wizard pop-up window size is 400 pixels tall by 600 pixels wide, but can be larger if needed. If you do resize the window, do so proportionately. For example, if you make the wizard 25% taller, you must also make it 25% wider.
  • A wizard pop-up window is resizable. If the wizard window is resized, the left pane (help/steps) and the right pane (content area) resize to maintain their original width ratios of 30 to 70, respectively.
  • The wizard window must be smaller than the application's main browser window.
  • The wizard window should open offset slightly to the right from the center of the screen.

 On wizard steps containing multiple entry fields, consider providing a Restore Defaults button. Do not provide a Restore Defaults button on pages with only one or two entry fields. The button must appear in the content area, above the Previous, Next, Finish, and Cancel buttons, not horizontally in line with them. Follow the guidelines for those buttons, in 7.1 Action Buttons.

 Do not provide Reset buttons in a wizard. This is just one particular case of the general prohibition of Reset buttons everywhere, in wizards and non-wizards, secondary content pages and primary content pages.

 Allow users to perform actions on the data presented in the wizard as long as those actions do not take the user out of the wizard. 

An example of a type of permissible action is allowing users to perform inline actions on objects in tables. The actual action is performed by clicking the Next or Finish button (not by making the inline selection). Another example of a permissible action is allowing the user to either select one of a number of existing objects or to create a new object by entering the name of that new object in the wizard.

An example of an impermissible action in a wizard is a "New ..." button on a table of objects that would take the user out of the context of the task they are currently performing and into the context of creating a new object.  

 Any buttons in addition to Previous, Next, Finish, and Cancel, must appear above those buttons in the content area, not horizontally aligned with the Previous, Next, Finish, or Cancel buttons. An example is the optional Restore Defaults button.

 Use headline capitalization for step titles on both the left and right panes.  Use sentence capitalization for contextual help text, and step instructions. See 2.5 Text Capitalization.   

 Step titles should be brief and to the point.  Examples include "Select LUNs," "Confirm Volume Mapping," and "Review Selections."

13.1.1 Steps List Anatomy

 When providing a steps list in a wizard, you must use the design described in this section.

 Put the steps list and contextual help each in its own tabbed pane in the left pane. Label the steps list tab "Steps," and make it the active (foreground) tab by default as shown below:

Display the current step's name in the steps list in a bold font. Prefix the step number with the arrow glyph () available from 2.4.3, Icons, Glyphs and Graphics Repository.

Make links of the visited steps' names in the steps list, if possible.

 If the user goes to a previous step, the names of all the steps after that one cease to be links. This is true regardless of how the user gets to that previous step (by clicking Previous button in the right pane or by clicking that previous step's name in the steps list).

Display all the steps' names in the steps list--previous, current, and future--unless any steps branch. "Branch" means that the user's input in a step determines which of several sequences of steps will be presented next. "Major step" means a step that has no parent step; a major step is any step that is not a substep as defined by 13.1.1.1 Branching Represented as Substeps. Follow this decision tree to find out how to display branching steps and their subsequent steps:

  1. For a steps list that does not branch: Display this step's name in the steps list. Do not display any steps as indented substeps of this step. Do not use a placeholder for the next step.
  2. For a list of steps that does branch:

    1. If this is a major step (not a substep), then:
      1. If any branch of this major step has its own branching step (in other words, a sub-substep), then you must represent this major step's branching with the design in 13.1.1.2 Branching Represented as Placeholder. This rule prevents sub-substep representations from occurring. The placeholder design represents the major step's branches as a placeholder for the undetermined steps. The placeholder is replaced by the actual steps after the branch fork is passed--after the branch path is chosen.  (You are now done following this decision tree.)
      2. If instead no branches of this major step have their own branching steps, then:

        1. If all branches of this major step converge on the next major step: In other words, the branching occurs only in the details within this step. Then you must use the design in 13.1.1.1 Branching Represented as Substeps. This design represents each branch as a hidden set of substeps that appears indented in the list only after the branch is determined by the user's input to the major step. This design is preferable to the Placeholder design (see next bullet in this decision tree), and should be sufficient for most branching wizards. (You are now done following this decision tree.)
        2. If instead the branches of this major step diverge at their ends, and the paths have no steps in common after the branch, then you must use the design in 13.1.1.2 Branching Represented as Placeholder. This design represents the branches as a placeholder for the undetermined steps. The placeholder is replaced by the actual steps after the branch fork is passed--after the branch path is chosen. This design may be used only if the substeps design (see previous bullets in this decision tree) cannot be used, because that substep design gives the user more information. (You are now done following this decision tree.)
    2. If instead this branching step is already represented as an indented substep as is defined by 13.1.1.1 Branching Represented as Substeps (that is, it is not a major step): Then you must abandon the substep design for the parent, major step of this branching substep. You must instead represent that parent, major step with a placeholder as described in 13.1.1.2 Branching Represented as Placeholder. Doing so will prevent this substep from being represented as an indented substep. (You are now done following this decision tree.)

13.1.1.1 Branching Represented as Substeps

This design represents each branch of a wizard's major step as a hidden set of substeps that appears in the list only after the branch is determined by the user's input to the major step.

 Use this substeps design for a major step in the steps list, in these conditions:

  • A major step branches. 
  • All branches of this step converge on the next step--the peer of this step that branches. In other words, the branching occurs only in the details within this step. (This design may not be used if the branches diverge; the placeholder design must be used instead. See 13.1.1.2 Branching Represented as Placeholder.)

 This design is preferable to the placeholder design (13.1.1.2 Branching Represented as Placeholder), and should be sufficient for most branching wizards.

In most cases where a major step branches, you can contain the variable steps as substeps of one major step. Substeps appear only as needed to accommodate selections made in a previous, branching, major step. Grouping these substeps under a parent step allows the user to more easily get an overview of the wizard, by displaying only the major steps when the wizard is initially displayed. The substeps appear in the steps list only when the user passes the branching point. The wizard has a corresponding page in the right pane for both major steps and substeps.

The following example sequence of three pictures illustrates the design of the substeps approach to handling branching. In the first image, the user selects either the QFS or SAM-FS file system type. If the user selects QFS, he must then specify which type of QFS. If the user selects SAM-FS, he does not need to specify another filesystem parameter and thus no substeps would appear. Therefore, the total number of steps that would appear in the left pane cannot be known until the user makes this choice. Using substeps allows the wizard to list all of the major steps (the flush left steps, numbered 1-7) and still allow for this unknown factor.

In the following picture, the user has selected QFS and has proceeded to substep 1.1. In the steps list, that first substep is bold and preceded by the arrow glyph.

In the following picture, because the user has selected "Dual-Allocation," he must then specify which percentages to allocate to each disk. Notice that in the list of steps, the second substep is now bold and is prefixed with the arrow glyph.

 Follow these guidelines for the substeps used to handle a branching major step.

  • For the major steps, follow the guidelines in 13.1.1 Steps List Anatomy.
  • Substeps are dynamic. They appear in the list of steps only if the user makes a selection that results in a substep.
  • All of the substeps that correspond to a major step appear in the list of steps below the parent step when the user clicks the Next button from that major step.
  • Display the substeps in the list of steps beneath the parent step, bold and left-aligned with the first letter of the parent step.
  • Prefix each substep with the arrow glyph as you do the major steps: ().
  • With regard to linking, follow the same guidelines for substeps as for major steps.
  • Users can click the Previous button in the content pane (the right pane) and step back through the substeps, just as they can major steps.

13.1.1.2 Branching Represented as Placeholder

This design represents each branch in the steps list as a placeholder for the undetermined steps before the branch is chosen. The placeholder is replaced by the actual steps after the branch path is chosen.

 Use the placeholder design only if all of these conditions are true:

  • A major step branches.
  • The substeps design cannot be used (13.1.1.1 Branching Represented as Substeps), because the branches diverge even at their ends, or because a substep would itself branch.

 This placeholder design is less desirable than the substeps design (13.1.1.1 Branching Represented as Substeps ), because this placeholder design provides the user with less information. The placeholder design is only a fallback for use when the substeps design cannot be used.

If a wizard's branch results in two sets of steps that are too divergent to be handled by substeps as described in 13.1.1.1 Branching Represented as Substeps , display only the current and past steps in the left pane. Display a placeholder after the last known step in the list to let the user know that additional steps will appear once the fork is passed.

This placeholder indicates to the user that additional steps will appear based on their choice or choices in the preceding step.

 When displaying a placeholder in the left pane, follow these guidelines:

  • In the list of steps, for the known steps, follow the guidelines in 13.1.1 Steps List Anatomy and 13.1.1.1 Branching Represented as Substeps.
  • In the list of steps, after the last known step, display a placeholder to indicate that additional steps will appear based on the user's choices in the preceding step. The placeholder must read "[Steps determined by choice(s) in the preceding step.]," but without the quote marks.
  • Once the user passes the branch, the placeholder disappears and the remaining steps are displayed in their entirety, as shown in the example pictures above.

13.1.2 Help in the Left Pane

 When the user clicks the Help tab in the left pane, contextual help for the current step is displayed by the Help tabbed pane becoming active (coming to the foreground). If the user clicks the Next button while the Help tab is selected, the Help tabbed pane displays the help text for that next step (rather than the left pane reverting to the Steps tab being active).

 Under no circumstances should a wizard's left pane help contain links to destinations outside of that help pane. 

See 11.3 Left-Pane Help, for more information about left pane help in wizards specifically. In wizards and other pop-up windows with help in their left pane, add inline help to the right pane only if the left pane help does not suffice. See 11.1.1 Deciding Whether to Provide Inline Help.

13.1.3 Overview Page

An Overview page provides a brief introduction to the wizard and its steps. The Overview page is typically needed only in very complex wizards or in wizards that don't immediately display a complete list of steps, but use the steps placeholder. Overview pages can help users determine whether a wizard meets their needs. In addition, an Overview page can inform users about the effects of using the wizard.

 Use an Overview page under any of these circumstances:

  • The wizard's left pane does not immediately display a complete list of steps, but uses the steps placeholder.
  • The wizard takes actions that might affect a user's system in unexpected ways.
  • The wizard is very complex and providing an overview of that complexity might help the user perform their task more easily.
  • The user may need to have certain pieces of information available to input into the wizard in order to perform this task. The overview page may act as a checklist of what to have prepared before the user begins their work.

Include these buttons in the Overview step: Previous, Next, and Cancel. (Disable the Previous button, because the Overview step is the first step.)

13.1.4 Review Selections Page

 For wizards with three or more user input steps, the Review Selections page is required.

 For wizards with only two user input steps, adding a third step for a Review Selections page is optional. If one or both of the steps are complex with a large amount of user input, the user might benefit from a review of the information they have provided. Conversely, the steps might be simple enough that a Review Selections page only adds an unnecessary step.

13.2 Behavior

This section of guidelines describes wizard behavior in these basic circumstances:

  • Multiple instances of a wizard
  • First pass through wizard
  • User did not backtrack to previous steps with the Previous button
  • Use of the Cancel button

More complex circumstances are addressed in the following sections:

  • 13.2.2 Backtracking
  • 13.2.3 Finish Button and View Results Page

In this section, the term "validation error" means an error on a user input control on one particular step. For example, a wizard for creating an object might have a field in which the user must enter a name for the new object, and the user might enter a name containing illegal characters. This would cause a validation error. 

The "validation errors" referenced in these guidelines include malformed individual entries, illegal combinations of entries in the same step, and values that are illegal based on the value of a previous step. Most required input controls require a non-blank, non-null value before they qualify as having a valid setting. Some required input controls accept blank or null values as valid, but in those cases the blank or null value is one of the choices that the user must explicitly make or enter, to distinguish them from the absence of user preference for a value.

Optional input controls need not have any value to qualify as valid.  If an optional control does have a value, that value must be legitimate.  The user can eliminate a validation error state in an optional control either by removing that value and leaving no value, or by changing the value to a legitimate one.

13.2.1 Basics

 Multiple instances of the same wizard should not be allowed.  If a user tries to launch a wizard that is already in progress, switch focus to that existing wizard.  If a wizard is running, do not disable the button that would launch the wizard but rather bring focus to the existing wizard.   Disabling the button may cause problems as it often becomes difficult to determine when the button should be re-enabled and can get "stuck" in a disabled state.

 Entries made in any step may only affect a wizard's future steps, never previous steps. That is, allow previous steps to influence later steps, but do not allow later steps to influence previous steps. In particular:

  • Do not change the identities or contents of previous steps because the user enters something in a later step.
  • Determine the validity of a user's entry in a step only on the basis of the entries in that and previous steps. Entries in later steps must not cause heretofore valid entries in previous steps to become invalid. If the user makes an entry in the current step, and that entry clashes with an entry in a previous step, mark only the current step's entry as invalid. In some cases this may indicate a flaw in the design of the wizard and the step sequence should be revisited.

 Check the validity of each input control as soon as the input focus leaves that control, if checking at that time makes sense. When it does not make sense, defer validity checking until the user clicks that step's Next or Finish button. Shifting the input focus away from a control is not a foolproof sign that the user thinks that control's value is final. Clicking the Next or Finish button is the only foolproof sign.

  • Check when focus changes, only if that control's value can be validated in isolation from the values of all other controls in that step. That's because the current values in the other controls might not yet be what the user intends.
  • Defer checking if the user might want to leave that control's setting partially complete while setting other controls in this or previous steps. Often users take advantage of their random access to all the controls in this and previous steps, to make their settings non-sequentially.
  • Defer checking if you are uncertain whether the user thinks they are done making a setting when they shift the focus.

When a wizard has a Review Selections step (which by definition contains the Finish button), all the user input is known before the Review Selections step is displayed so all validations and confirmations should be done before the Review Selections step is displayed.  However, simple two-step wizards may not always have a Review Selections step. Consequently, they have input controls on the same page as the Finish button so clicking the Finish button may be the system's first signal that it is time to validate or confirm the input.

In some cases the wizard cannot accurately predict validity of the input until the wizard's actions actually execute upon the Finish button's click. One cause is that the system conditions have changed in the time lag between the preliminary validation and the click of the Finish button. Another cause is that validation depends on responses from system components that cannot respond to hypothetical actions.

 When the user clicks the Next or Finish button on a step's page:

  • Immediately check the validity of that step's input, if possible.
  • If the current step's input controls have validation errors, keep that page displayed while showing the usual validation error messages. Do not proceed to the next page, do not jump to the View Results page, and do not close the wizard. In other words, treat each step page as you would a non-wizard page, following the guidelines in 14.2 JavaScript Alerts, regardless of whether a step page has a Next button or a Finish button.
  • If there are no validation errors on that step then either display the next step if the Next button was clicked, or follow the guidelines in 13.2.3 Finish Button and View Results Page if the Finish button was clicked.

 When the Next button is clicked, do not validation check the next step's page before displaying it.

 When the user clicks the Finish button, optionally display an alert to inform them that they are about to commit an action that is irreversible, for example, the deletion or reconfiguration of an object or objects. Use such an alert with care. Understandably, users might be irritated if constantly asked to confirm the same actions. In some cases, inline text that informs the user of the impending action is sufficient, but usually a JavaScript alert window is more appropriate. Follow the guidelines in 14 Alerts and Messages.

 If the application session times out, the wizard window remains open. However, the entire window's contents are replaced with a full-page alert telling the user that the session has timed out. The full-page alert contains a Close button on the bottom right. This guideline for wizards is just one particular case of the more general guideline for all pop-up windows. See 3.2 Pop-Up Windows.

 When displaying error messages, give users error messages that offer fixes.  For example, provide an error message with a button for each of the several possible ways for fixing the error.  If that is not possible, then at least tell users how to fix the problems, by offering concrete, specific action advice in error and warning messages.  For example, tell users exactly which step had the problem, and offer suggestions for fixing it. One example of such an error message might read, "In step 2, the value you have entered for Capacity is too high to be supported by the virtual disks you have selected in step 3. To fix this problem, either enter a lower Capacity value in step 2 or select additional virtual disks in step 3".

 When a user selects the Cancel button, display a pop-up confirmation only if the wizard is complex enough to warrant concern that data might be lost.  If the wizard is simple, a pop-up confirmation is not necessary.  In either case (complex or simple wizards), do not display a confirmation if the wizard is closed during the first step. 

13.2.2 Backtracking

This section supplements 13.2.1 Basics, to cover what happens when the user backtracks to previous steps by clicking the Previous button, and moves forward again with the Next button.

Additional and more complex circumstances are addressed by the later section 13.2.3 Finish Button and View Results Page.

 If the user clicks the Previous button, display the previous step even if the current step has validation error indicators or undetected validation errors. Clicking the Previous button does not cause a validation check of the current step.

 Visiting previous steps shows those steps with all their values intact from the last visit in this session of wizard use, regardless of whether any changes have been made in subsequent steps.

 User entries are saved from page to page. If the user enters information on a wizard's page and clicks Next, Previous, or a visited step, then that information on the page being left is saved implicitly.

 Changing a previous step's values and clicking that step's Next button requires the same wizard branching decisions that must be made if this were the first pass through the wizard. Consequently, the sequence of the subsequent steps might change. The following two guidelines apply to two circumstances:

  1.  If the previous step's change does not cause the sequence of subsequent steps to change, then the subsequent steps must retain the values they had when last visited, even if those values are now invalid. 
    • Invalid values will be flagged as erroneous as soon as each subsequent step appears (see guideline farther below).
  2.  If instead the previous step's change causes the sequence of subsequent steps to change, consider replacing all the values in those subsequent steps with default values, making this re-traversal unfold as if the user had never before been to these steps. The reversion to default values happens even if the user's previously entered values would still be valid. The reversion begins with the first step whose identity differs from the step that was in that sequence spot before the change. Once a step reverts to its default values, all steps after that one also revert to default values.
    • This recommendation is strongest if the new sequence of steps qualifies as a drastically new context for the user deciding on the values.
      • That can be true even if the component steps of that sequence happen to be the same as before, and so merely are in a different order. The inconvenience of reentering some of the same values is outweighed by reducing the risk of the user blithely accepting prior values that are no longer appropriate. Values might be inappropriate from the user's point of view even if they are valid from the system's point of view. So the system's validity checking is not sufficient to warn the user about what values should be reconsidered.
    • This recommendation is weakest if the change in sequence makes no difference in the desirability of the prior values.
      • For example, suppose the only change in a sequence is elimination or injection of one step, and that that step's values have no consequence for the desirability of the values in the remaining steps. If the user had spent a lot of time entering those subsequent steps' values in the prior traversal, the user would be inconvenienced at having to reenter them.

 Clicking a previous step name that is a link in the steps list, is equivalent to clicking the sequence of Previous buttons in the right pane to get to that step. The same guidelines apply.

 When a previous step is displayed, all the subsequent step names in the steps list become unavailable for clicking and do not have the link appearance. This happens regardless of how the user gets to a previous step, either by clicking its name in the steps list or by clicking the Previous button.

13.2.3 Finish Button and View Results Page

This section supplements 13.2.1 Basics and 13.2.2 Backtracking, to cover what happens when the Finish button is clicked.

 If the Finish button's page also has other user input controls, then when the Finish button is clicked first follow the guidelines above for validation checking and error displaying when a Next or Finish button is clicked. Only then should the wizard execute.  Be sure to follow the guidelines below regarding the Finish button.

 When the user clicks the Finish button, optionally display an alert to inform them that they are about to commit an action that is irreversible; for example, the deletion or reconfiguration of an object or objects. This alert should be displayed immediately after the Finish button is clicked but before the wizard's action is initiated. Use such an alert with care as users might become irritated if repeatedly asked to confirm the same actions. In some cases, inline text that informs the user of the impending action is sufficient but usually a JavaScript alert window is more appropriate. Follow the guidelines in 14 Alerts and Messages.

 If a request for confirmation of the wizard's action is displayed after the Finish button is clicked, display the confirmation message in a self-contained secondary-content page, not as part of the wizard. For a wizard that is in a pop-up window the message should appear in a JavaScript window.

 After the Finish button is clicked, if a progress indicator is justified display it in the Results page of the wizard.

  • This is for "synchronous" actions, for which it makes sense to leave the wizard displayed while the action is being executed by the system.
  • If instead an action will take so long that the user will be bothered by the wizard remaining displayed, that is an "asynchronous" action. Asynchronous actions are finished as far as the wizard is concerned, as soon as those actions are successfully begun, such as jobs submitted. Subsequent progress of asynchronous actions such as jobs must not be shown in the wizard, but instead in a separate progress indicator.

Non-wizards need other places to display progress indicators.  But wizards have a ready-made place for progress indicators, so you should consider using it instead of creating another place.

 If the wizard did not perfectly execute after the Finish button was clicked, then display the error messages in the View Results page. But:

  • This is for "synchronous" actions, for which it makes sense to leave the wizard displayed while the action is being executed by the system, so that failure messages can be displayed in the View Results page.
  • If instead an action will take so long that the user will be bothered by the wizard remaining displayed, that is an "asynchronous" action. Asynchronous actions succeed as far as the wizard is concerned, as long as those actions are successfully begun, such as jobs successfully submitted. Subsequent failure of asynchronous actions must not be shown in the wizard, but elsewhere such as a separate progress indicator.

 A given wizard within a given product should either always have a View Results page or never have a View Results page, regardless of that wizard's success or failure.
